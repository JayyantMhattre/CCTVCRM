# Database Future Considerations

**Project:** Aarvii CCTV AMC Management System
**Phase:** D0-4 — Entity Model, ER Diagram & Database Architecture (design only)

This document shows how the **current frozen design** already leaves room for likely future needs — **without changing the current design** and without implementing anything out of scope (freeze §21). Every item below would still require an approved change request (freeze §22) before any build.

---

## 1. Future extensibility (general)

| Built-in headroom | How it helps later |
|-------------------|--------------------|
| Schema-per-module + logical cross-schema references | Any domain can be extended or extracted to its own service/database without touching others |
| UUID keys generated by owning modules | No id collisions when splitting databases or merging branch data |
| Append-only histories (status histories, activities, approvals) | New analytics/SLA features can be derived from existing data — no retro-instrumentation |
| Frozen status vocabularies in CHECK constraints | New statuses are additive constraint changes via change request, not redesigns |
| Platform Files/Audit/Webhooks reuse | New document types, audit events, and outbound integrations need zero schema change |

## 2. Spare replacement billing

**Already supported by Option B (approved in D0-4):**

- `invoice_type = 'SpareReplacement'` exists today.
- The invoice can reference the originating `ticket_id` / `service_visit_id` (nullable columns already present).
- `InvoiceLine` rows describe replaced parts (description, quantity, unit price) without any parts-catalog dependency.

**Future (change request):** a parts/spares catalog could be referenced from `InvoiceLine` by adding a nullable logical reference column — no change to the current invoice model.

## 3. Additional service billing

**Already supported:** `invoice_type ∈ {EmergencyService, AdditionalCharges, Other}` with free-form lines covers emergency callouts and extra work beyond included services.

**Future (change request):** a priced service-catalog (rate card) could back `InvoiceLine` via an optional reference; AMC plan versions already isolate included-vs-billable services per term, so no contract redesign is needed.

## 4. Multi-branch growth

**Headroom in current design:**

- All aggregates hang off Customer/Site — a future `branch_id` (logical reference) on operational roots (Lead, Site, ServiceSchedule, Ticket, Invoice, Engineer) is **additive**, nullable-first, and backfillable.
- Business numbers are prefix-based (`LD-2026-0001`); a branch segment can extend the format without breaking uniqueness.
- The platform is multi-tenant by design; a branch dimension can also be modeled with platform tenancy if branches become separate operating entities — no current table prevents either path.

**Explicitly not built now:** no branch tables, no branch columns — single-operation V1 per the freeze.

## 5. Advanced reporting

**Headroom in current design:**

- Reporting owns no schema (reads via module queries), so a future read-model/warehouse (replica, materialized views, or export pipeline) can be added without touching write models.
- Append-only histories give event-grain data for SLA, turnaround-time, engineer-productivity, and renewal-rate analytics.
- All lifecycle events already flow to the platform Audit store and are webhook-eligible — an analytics consumer can subscribe without schema change.

**Future (change request):** dedicated reporting schema / OLAP store fed by outbox events.

## 6. Future payment gateway support

**Out of scope in V1** (freeze §21) — `Paid` is a manual admin status update.

**Headroom in current design:**

- `InvoiceStatusHistory` already records who/when for payment transitions.
- A future `payment` entity (gateway reference, transaction id, method, amount) would be a **new table referencing `invoice_id`** — zero change to the existing Invoice aggregate; the `Sent → Paid` transition simply gains an automated trigger source.
- Amount columns are `numeric(18,2)` end-to-end — gateway-safe precision.

## 7. Future inventory integration

**Out of scope in V1** (freeze §21: Inventory Stock Management, Purchase Orders, Vendor Management).

**Headroom in current design:**

- Asset tracking is deliberately summary-only (freeze §7); an inventory module would be a **separate business module with its own schema**, linked logically by `site_id` / `invoice_line` references.
- Spare-replacement invoices (Option B) already capture consumption events that an inventory module could later reconcile against.
- Because cross-schema references are logical, adding an inventory schema introduces **no FK churn** in existing CCTV schemas.

---

## Guardrails

1. **Nothing in this document is implemented in V1.** It documents headroom, not commitments.
2. Any item above requires an approved change request (freeze §22) and, where architectural, an ADR per platform governance.
3. Additions must follow the same rules as the current design: schema-per-module ownership, logical cross-schema references, FileId-only file references, platform audit reuse, no Core Platform changes.
